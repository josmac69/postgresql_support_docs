# The PostgreSQL Recovery Process: Source-Code-Level Architecture and Version-by-Version Evolution, PG11 → PG19

## TL;DR
- PostgreSQL recovery is one machine — `StartupXLOG()` in `xlog.c` orchestrating `InitWalRecovery()` → `PerformWalRecovery()` → `FinishWalRecovery()`/`EndWalRecovery()` in `xlogrecovery.c` — that serves crash recovery, archive/PITR, and standby replay; the differences are which sources feed `ReadRecord()`, where replay stops, and what marks the end (an end-of-recovery checkpoint for crash recovery, an end-of-recovery WAL record for timeline-forking promotion).
- The largest structural changes across PG11→PG19 were: `recovery.conf` → GUCs + signal files (PG12), WAL prefetching for recovery plus the split of `xlog.c` into `xlogrecovery.c`/`xlogprefetcher.c` (PG15), logical decoding on standby (PG16), WAL summarization/incremental backup and slot synchronization (PG17), the async-I/O subsystem (PG18), and PG19's operational hardening of AIO plus first-class recovery observability (`pg_stat_recovery`, `WAIT FOR`).
- For DBAs the practical arc is faster and more observable recovery/failover: `recovery_init_sync_method=syncfs` (PG14), `recovery_prefetch` (PG15), `pg_stat_recovery_prefetch`/`log_recovery_conflict_waits` monitoring, restartpoint counters in `pg_stat_checkpointer` (PG17), and PG19's `pg_stat_recovery`/`WAIT FOR LSN` — but PG18/PG19 async I/O still does **not** cover the WAL redo read path, which remains posix_fadvise-based.

## Key Findings
- Recovery is coordinated by the **startup process** (`StartupProcessMain` in `startup.c` → `StartupXLOG()` in `xlog.c`). The low-level replay logic lives in `xlogrecovery.c` (added PG15), WAL reading in `xlogreader.c`, redo helpers in `xlogutils.c`, prefetching in `xlogprefetcher.c` (added PG15).
- The redo point is determined from `pg_control` (`global/pg_control`) plus an optional `backup_label`. Replay runs from the checkpoint's REDO LSN forward, applying records via per-resource-manager `rm_redo` handlers, skipping records whose LSN ≤ the page's `pd_lsn` (redo of non-FPI blocks is not idempotent).
- Crash recovery ends with an **end-of-recovery checkpoint**; archive recovery/promotion that forks a timeline writes an **end-of-recovery WAL record** (`XLOG_END_OF_RECOVERY`) rather than requiring an immediate checkpoint. Invalid-page tracking (`XLogCheckInvalidPages()` in `xlogutils.c`) PANICs with "WAL contains references to invalid pages" if referenced-but-missing pages are not resolved by a later DROP/TRUNCATE before consistency.
- Version deltas cluster into: configuration model (PG12), retention/observability (PG13), startup speed and conflict logging (PG14), recovery prefetch + refactor + archive modules (PG15), logical decoding on standby + pg_stat_io (PG16), WAL summarizer/incremental backup + slot sync + restartpoint counters (PG17), async I/O (PG18), and AIO operability + recovery observability + `WAIT FOR` + dynamic wal_level (PG19).

## Details

### Part 1 — Recovery architecture, in depth

#### 1.1 Entry point and process model
The startup process is launched by the postmaster and runs `StartupProcessMain()` (`src/backend/postmaster/startup.c`), which calls **`StartupXLOG()`** (`src/backend/access/transam/xlog.c`). The file header of `xlog.c` documents the split: `xloginsert.c` builds WAL records, `xlogrecovery.c` handles "WAL recovery and standby code," `xlogreader.c` is the "facility for reading WAL files and parsing WAL records," and `xlogutils.c` holds "helper functions for WAL redo routines." Per the source comment, `StartupXLOG()` "coordinates database startup, performing WAL recovery, and the transition from WAL recovery into normal operations."

`StartupXLOG()` reads `pg_control` into memory, and if the state is not a clean "shut down" (e.g. `DB_IN_PRODUCTION`), recovery is required. On crash, before replay the startup process (a) removes stray temporary WAL segments under `pg_wal`, and (b) synchronizes the entire data directory (see `recovery_init_sync_method`, PG14). It then calls the `xlogrecovery.c` functions:

- **`InitWalRecovery(ControlFile, ...)`** — per the source comment, this "analyzes the control file and the backup label file, if any, and figures out whether we need to perform crash recovery or archive recovery, and how far we need to replay the WAL to reach a consistent state." It sets `InRecovery`, establishes tablespace symlinks from `tablespace_map`, and locates/validates the checkpoint record. It "doesn't yet change the on-disk state, except for creating the symlinks."
- **`PerformWalRecovery()`** — the core redo loop.
- **`FinishWalRecovery()` / `EndWalRecovery()`** — cleanup, computing the end-of-WAL position (`EndOfWalRecoveryInfo`), and shutting down the prefetcher/reader (`XLogPrefetcherFree()`, `XLogReaderFree()`).

Note on global-variable ambiguity (which the refactor authors themselves flagged): `InRedo`, `InRecovery`, `InArchiveRecovery`, and `RecoveryInProgress()` all mean subtly different things — `InArchiveRecovery`/`ArchiveRecoveryRequested` distinguish archive/PITR/standby from plain crash recovery; `InRecovery` is set while the startup process is actively replaying; `RecoveryInProgress()` is the shared-memory flag other backends test.

#### 1.2 Reading pg_control and determining the redo point
`pg_control` stores the latest checkpoint LSN, its REDO LSN, `nextXid`, `nextOid`, `nextMulti`, `oldestXid`, `minRecoveryPoint`, `minRecoveryPointTLI`, `backupStartPoint`, `backupEndPoint`, timeline IDs, and the database state enum. At startup PostgreSQL reads the latest checkpoint record from the WAL segment named by `pg_control`; the checkpoint's **REDO location** (printed after "redo" in `pg_controldata`) is where replay begins — it corresponds to the WAL insert position at the moment the checkpoint began, so it is normally earlier than the checkpoint record's own LSN. (If `pg_control` is corrupt, the design allows scanning existing WAL segments newest-to-oldest to find the latest checkpoint.)

If a `backup_label` is present (restore from base backup), the redo point comes from the label ("starting backup recovery with redo LSN … checkpoint LSN … on timeline …"), not from `pg_control`. Since PG12, if `backup_label` exists but neither `recovery.signal` nor `standby.signal` is present, recovery uses the checkpoint in `backup_label` for crash recovery — a common misconfiguration trap when migrating from `recovery.conf`, since a self-contained `pg_basebackup --wal-method=stream` backup will recover only to the end of the backup and then stop.

#### 1.3 The replay loop (`PerformWalRecovery` / `ApplyWalRecord` / `ReadRecord`)
`PerformWalRecovery()` initializes shared progress trackers (`lastReplayedReadRecPtr`, `lastReplayedEndRecPtr`, `lastReplayedTLI` in `XLogRecoveryCtl`), notifies the postmaster (`PMSIGNAL_RECOVERY_STARTED`), reads the first record with **`ReadRecord()`**, and enters the loop calling **`ApplyWalRecord()`** per record. `ReadRecord()` drives the `XLogReaderState` (`xlogreader.c`) via a `page_read` callback (`XLogPageRead`) that fetches pages from the current source (`XLOG_FROM_ARCHIVE`, `XLOG_FROM_PG_WAL`, `XLOG_FROM_STREAM`).

For each record, `ApplyWalRecord()`:
1. Checks recovery-target stop conditions (`recovery_target_*`, honoring `recovery_target_inclusive`).
2. Dispatches to the resource manager's redo routine via `GetRmgr(record->xl_rmid).rm_redo`. RMgrs include Heap, Heap2, Btree, GIN, GiST, SP-GiST, Hash, XLOG, Transaction, Storage, CLOG, MultiXact, Standby, Sequence, and others.
3. Buffer redo (`XLogReadBufferForRedoExtended` in `xlogutils.c`) reads the target page and compares the record LSN to `pd_lsn`. If the WAL record carries a **full-page image (FPI)** — written on first modification after a checkpoint when `full_page_writes=on` — the page is restored wholesale; otherwise the delta is applied only if `record LSN > pd_lsn`. As Hironobu Suzuki's InterDB internals notes describe, this LSN gating is why non-FPI redo must run in order and exactly once (it is not idempotent); with `full_page_writes=on`, every post-checkpoint page's first change is a full replay.
4. Periodically calls `HandleStartupProcInterrupts()` so the loop can respond to config reloads, promotion triggers, and shutdown.

`CheckRecoveryConsistency()` is called during replay; once `minRecoveryPoint`/`backupEndPoint` is passed and invalid-page checks pass, it sets `reachedConsistency = true`, sends `PMSIGNAL_RECOVERY_CONSISTENT` / `PMSIGNAL_BEGIN_HOT_STANDBY`, and (on a standby with `hot_standby=on`) read-only connections become possible ("consistent recovery state reached at …").

#### 1.4 Full-page writes, invalid pages, and relation truncation during redo
Invalid-page tracking: when redo references a page in a relation/block that does not exist on disk (e.g. a block beyond EOF), the page is remembered in a hash table rather than immediately erroring, on the expectation that a later WAL record (DROP TABLE, TRUNCATE, or a smaller-file truncation) will explain it. `XLogCheckInvalidPages()` (`xlogutils.c`) runs at the consistency point; if unresolved references remain it raises `PANIC: WAL contains references to invalid pages`. A historical bug class here: after promotion, `minRecoveryPoint` is only cleared at the first post-promotion checkpoint, so a crash immediately after promotion could run crash recovery with a stale `minRecoveryPoint` and hit this PANIC before seeing the explaining TRUNCATE record. Redo of `smgr`/storage truncation shrinks relation files during replay; the prefetcher must suppress prefetch for blocks past a pending truncation (logged as "suppressing prefetch in relation … until … is replayed, which truncates the relation").

#### 1.5 End of recovery: checkpoint vs. record
After replay stops, cleanup differs by mode:
- **Crash recovery** (no timeline fork): request an **end-of-recovery checkpoint** (logged as "checkpoint starting: end-of-recovery immediate wait"). The checkpointer performs it; the control file is then updated to `DB_IN_PRODUCTION`.
- **Archive recovery / promotion** that forks to a new timeline: write an **end-of-recovery WAL record** (`XLOG_END_OF_RECOVERY`, carrying `ThisTimeLineID`/`PrevTimeLineID`) instead of forcing a full checkpoint, then request a checkpoint asynchronously. On promotion, cascading standbys are notified (`WalSndWakeup`), and `RemoveNonParentXlogFiles()` cleans WAL that doesn't belong to the new timeline.

Timeline handling: `checkTimeLineSwitch()` validates timeline switches; a promotion increments the timeline ID and writes a `.history` file. `recovery_target_timeline` (`current`/`latest`/numeric) selects which timeline to follow; `latest` is the default and is essential for standbys following a promoted primary.

#### 1.6 Archive recovery and PITR machinery
- `restore_command` fetches archived segments; `archive_cleanup_command` runs at each restartpoint; `recovery_end_command` runs once at end of recovery (`ExecuteRecoveryCommand()`).
- Recovery targets: `recovery_target='immediate'`, `recovery_target_time`, `recovery_target_xid`, `recovery_target_lsn`, `recovery_target_name` (from `pg_create_restore_point()`). At most one may be set (`validateRecoveryParameters()`), else an error is raised ("At most one of recovery_target, recovery_target_lsn, recovery_target_name, recovery_target_time, recovery_target_xid may be set").
- `recovery_target_action`: `pause` (default; resumable via `pg_wal_replay_resume()`), `promote`, or `shutdown`. If `hot_standby` is off, `pause` behaves like `shutdown`. With `shutdown`, `recovery.signal` is **not** removed, so a subsequent start re-shuts-down unless reconfigured.
- Signal files (PG12+): `recovery.signal` → archive recovery (targeted), `standby.signal` → standby mode. They are removed when recovery ends (except with the `shutdown` action). Backup-label-based restore without either signal file errors: `FATAL: could not find recovery.signal or standby.signal when recovering with backup_label`. Requesting `recovery.signal` without a `restore_command` errors with `FATAL: must specify restore_command when standby mode is not enabled` (`validateRecoveryParameters()`, behavior dating to commit 66ec2db).

#### 1.7 Standby / replication recovery
In standby mode the loop never stops at end of WAL; instead it fetches more via `restore_command`, then local `pg_wal`, then streaming. The **walreceiver** (started by the startup process) streams from `primary_conninfo`; `pg_stat_wal_receiver` shows its state, and a large gap between `pg_last_wal_replay_lsn()` and its `flushed_lsn` indicates WAL is arriving faster than it can be replayed. Hot standby builds a standby snapshot from a `RUNNING_XACTS` (`xl_running_xacts`) record: `standbyState` transitions `STANDBY_INITIALIZED` → (snapshot pending) → `STANDBY_SNAPSHOT_READY`. `RecordKnownAssignedTransactionIds()` maintains the `KnownAssignedXids` array; historically a heap-insert record seen before the first running-xacts record could overflow it (`FATAL: too many KnownAssignedXids`), which is why recovery writes the running-xacts record immediately before the checkpoint record.

Recovery conflicts arise when replay needs to remove rows / lock objects still visible to standby queries; `max_standby_streaming_delay` / `max_standby_archive_delay` bound the wait before cancelling queries (`ERROR: canceling statement due to conflict with recovery`); `hot_standby_feedback` pushes the standby's xmin to the primary to prevent premature cleanup (at the cost of primary bloat). `recovery_min_apply_delay` intentionally delays commit-record replay (only commit records are delayed; MVCC visibility keeps this safe). During recovery, the startup process holds a virtual transaction in `pg_locks` that owns all replayed `AccessExclusiveLock`s.

Restartpoints are the standby analogue of checkpoints: the checkpointer performs them at replayed checkpoint records (`CreateRestartPoint()`), forcing state to disk and updating `pg_control` so already-processed WAL need not be re-scanned. They cannot occur more frequently than checkpoints on the primary, and only at checkpoint records — so `max_wal_size` can be exceeded during recovery by up to one checkpoint cycle. `archive_cleanup_command` runs after each restartpoint (`%r` = last valid restart segment).

#### 1.8 Supporting machinery
- Checkpointer and background writer run during recovery (restartpoints + block cleaning, including hint-bit updates on the standby). Autovacuum is not active until recovery ends.
- WAL prefetching (`xlogprefetcher.c`, PG15+): `XLogPrefetcher` decodes ahead of replay via an `LsnReadQueue`, issuing `posix_fadvise` for referenced blocks not in shared buffers, bounded by `maintenance_io_concurrency` and `wal_decode_buffer_size`. It skips FPI blocks (`skip_fpw`), already-cached blocks, and blocks in soon-to-be-created/truncated relations, using a small recent-block window (`XLOGPREFETCHER_SEQ_WINDOW_SIZE`).
- Monitoring: `pg_stat_recovery_prefetch` (PG15), `pg_stat_wal` (PG14), `pg_stat_io` (PG16, tracks startup-process reads/writes), `pg_last_wal_replay_lsn()`, `pg_last_xact_replay_timestamp()`, `pg_stat_checkpointer` restartpoint counters (PG17), `pg_stat_recovery` (PG19). The cumulative statistics system is active on standbys, but WAL replay does not increment relation/database counters or `pg_statio_` views.

### Part 2 — Version-by-version changes

#### PostgreSQL 11 (baseline, 2018)
- **Configurable WAL segment size at initdb** (`--wal-segsize`, replacing the `--with-wal-segsize` configure option): a power of 2 between 1 and 1024 MB, default 16 MB (author Beena Emerson; reviewers included Andres Freund, Michael Paquier, David Steele, Robert Haas, Peter Eisentraut, Kuntal Ghosh, Tushar Ahuja). The size is read from the control file (`wal_segment_size`), so standbys inherit it and `pg_upgrade` can change it; `pg_basebackup` gained a `SHOW_WAL_SEGSZ` replication command. Operational impact: larger segments reduce file churn and archiving frequency on high-write systems; recovery reads/prefetches in the same segment granularity.
- Baseline for the recovery machinery described above: `recovery.conf` still used; `trigger_file`/`standby_mode` still present; logical replication of TRUNCATE rounds out the baseline feature set.

#### PostgreSQL 12 (2019) — the configuration revolution
- **`recovery.conf` removed** (authors Fujii Masao, Simon Riggs, Abhijit Menon-Sen, Sergei Kornilov). All recovery/standby parameters became normal GUCs in `postgresql.conf`, settable via `ALTER SYSTEM` → `postgresql.auto.conf`. Presence of `recovery.conf` now causes `FATAL: using recovery command file "recovery.conf" is not supported`.
- **Signal files**: `standby.signal` (standby mode) and `recovery.signal` (targeted archive recovery) replace `standby_mode`. `standby_mode` removed; `trigger_file` renamed `promote_trigger_file`. On promotion `standby.signal` is deleted (not renamed to `.done`).
- `recovery_target_*` are GUCs (currently PGC_POSTMASTER). `pg_basebackup -R` now appends settings to `postgresql.auto.conf` and creates `standby.signal`. **`pg_promote()`** SQL function added.
- DBA impact: backup/HA tooling had to be rewritten; the "last GUC read wins" model means replication settings should be appended to `postgresql.auto.conf` to guarantee they're read last. Recommendation: comment out stale `recovery_target_*` after recovery to avoid surprises on later restarts.

#### PostgreSQL 13 (2020)
- **`wal_keep_segments` → `wal_keep_size`** (in megabytes; author Fujii Masao). Migration: `wal_keep_size = wal_keep_segments × wal_segment_size` (typically 16 MB).
- **`max_slot_wal_keep_size`** (Kyotaro Horiguchi): caps WAL retained by replication slots; slots that would exceed it are marked invalid (visible in `pg_replication_slots`). This is a major operational safety valve against unbounded `pg_wal` growth from a dead standby's slot.
- **`ignore_invalid_pages`** developer GUC (Fujii Masao): lets recovery continue past references to invalid pages instead of PANICking — a last-resort tool that risks silently ignoring corruption.
- **Standby promotion can cancel a requested pause** (Fujii Masao): previously promotion could not proceed while the standby was paused.
- **Error if recovery target is not reached** (Leif Gunnar Erlandsen, Peter Eisentraut): previously a standby would promote itself at end-of-WAL even if the target wasn't reached — now it errors, preventing silent under-recovery.
- WAL usage tracking in EXPLAIN/autovacuum/pg_stat_statements (Kirill Bychik, Julien Rouhaud). `wal_level=minimal` WAL-skip for relation create/rewrite generalized via `wal_skip_threshold` (Kyotaro Horiguchi), fixing a prior crash-recovery data-loss bug. Faster replay of DROP DATABASE with many tablespaces (Fujii Masao) and faster large-relation truncation (Kirk Jamison).

#### PostgreSQL 14 (2021)
- **`recovery_init_sync_method`** (`fsync`|`syncfs`; author Thomas Munro): since commit 2ce439f3, crash recovery opens and fsyncs every file in the data directory, which can take minutes or hours with many files. `syncfs` (Linux only) synchronizes the whole filesystems containing the data dir, WAL, and tablespaces instead, avoiding per-inode faulting. Default stays `fsync`. Large win for startup latency on clusters with millions of files; caveats: syncfs also flushes unrelated data on shared filesystems, and it does not repair a cluster previously run with `fsync=off` or copied via unsynced rsync.
- **`remove_temp_files_after_crash`** (default `on`; author Euler Taveira, committed by Tomas Vondra): removes leftover `pgsql_tmp` files at restart after a backend crash. Caveat raised by AWS/RDS: with millions of temp files, single-threaded cleanup can significantly delay startup.
- **`log_recovery_conflict_waits`** (Bertrand Drouvot et al.): logs when recovery waits longer than `deadlock_timeout` for a conflict, printing the blocking PID and the pg_waldump-style WAL record context — greatly improves diagnosis of stalled replay on standbys.
- **Hot-standby KnownAssignedXids cleanup** improved (Simon Riggs, Michail Nikolaev): insufficiently aggressive trimming caused poor performance with large `max_connections` on the standby.
- **`pg_stat_wal`** view added for WAL generation stats. Startup-process interrupt handling hardened. (Note: an early `recovery_prefetch` implementation was committed this cycle as commit 1d257577 but was subsequently withdrawn via commit c2dc1934, deferring the feature to PG15.)

#### PostgreSQL 15 (2022) — recovery prefetch + the big refactor
- **`recovery_prefetch`** (`off`|`on`|`try`, default `try`; author Thomas Munro, reintroduced via commit 1d257577 "Optionally prefetch referenced data in recovery"): the `XLogPrefetcher` reads ahead in the WAL and issues `posix_fadvise` for referenced blocks not yet in shared buffers, reducing I/O stalls during redo/replay. Enabled only where `posix_fadvise` exists; bounded by `maintenance_io_concurrency` and `wal_decode_buffer_size` (default 512 kB, `Min: 64kB, Max: INT_MAX`, set only at server start). The commit message noted that "Better mechanisms for asynchronous I/O are possible in later work." (Docs also clarify `try` "enables the feature on systems where posix_fadvise is available.")
- **`pg_stat_recovery_prefetch`** view: columns `prefetch, hit, skip_init, skip_new, skip_fpw, skip_rep, wal_distance, block_distance, io_depth`. A subsequent patch addressed useless fadvise calls when a recent FPI would load the page into the buffer anyway (a real incident resolved by disabling `recovery_prefetch`).
- **`xlog.c` split into `xlogrecovery.c`** (author Heikki Linnakangas): recovery/standby code (`InitWalRecovery`, `PerformWalRecovery`, `EndWalRecovery`, recovery-target logic) moved out; `xlog.c` retains startup orchestration and checkpointing. `xlogprefetcher.c` was added. This is the file layout that Part 1 describes.
- **Archive modules / `archive_library`** (author Nathan Bossart): C modules as an alternative to `archive_command`, with a `contrib/basic_archive` example that durably fsyncs and atomically renames (via a temporary `archtemp.*` file) and compares an existing target file for idempotency. More reliable and potentially more performant than shell archiving; the library is loaded only by the archiver process.
- **`pg_walinspect`**: SQL functions to inspect WAL records (an in-database companion to `pg_waldump`).

#### PostgreSQL 16 (2023)
- **Logical decoding on standby** (Bertrand Drouvot, Andres Freund, Amit Khandekar; core commit 0fdab27ad68a059a1663fa5ce48d76333f1bd74c, committed by Andres Freund): logical replication slots can be created and consumed on a physical standby (requires `wal_level=logical` on primary and standby). Slots are invalidated if the primary lowers `wal_level` (infrastructure added in commit 26669757b6a). Slot creation waits for an `xl_running_xacts` record; the new **`pg_log_standby_snapshot()`** function on the primary accelerates it when the primary is idle. A subscriber can keep following the stream seamlessly across a standby promotion. Recovery-conflict handling was extended to invalidate conflicting logical slots, with extra info added to WAL records for slot-conflict handling.
- **`pg_stat_io`**: cluster-wide I/O by backend type/object/context, including the startup process's reads during recovery — key new observability for redo I/O behavior (e.g. whether `recovery_prefetch` is actually saving reads).
- Log additions: checkpoint and REDO LSN in `log_checkpoints` (Bharath Rupireddy, Kyotaro Horiguchi); new wait events.

#### PostgreSQL 17 (2024)
- **WAL summarizer + incremental backup** (author Robert Haas; walsummarizer commit 174c480508ac25568561443e6d4a82d5c1103487): a new `walsummarizer` background process (`summarize_wal=on`, retention via `wal_summary_keep_time`, default 10 days) writes block-modification summaries to `pg_wal/summaries`. `pg_basebackup --incremental` (`-i`) uses them; `pg_combinebackup` reconstructs a full copy; the new `pg_walsummary` tool inspects them; SQL introspection via `pg_available_wal_summaries()`, `pg_wal_summary_contents()`, `pg_get_wal_summarizer_state()`. Recovery relevance: incremental restore still requires the full recovery pipeline (`recovery.signal`/`standby.signal` + WAL replay) after `pg_combinebackup`; on a standby, incremental-backup replay begins from the most recent restartpoint (so a low-activity standby can fail an incremental backup if no new restartpoint has been created).
- **Slot synchronization / failover slots** (`sync_replication_slots`, `pg_sync_replication_slots()`; Hou Zhijie, Shveta Malik, Ajin Cherian, Peter Eisentraut): a physical standby synchronizes logical failover slots (marked with the new `failover` property) from the primary, so logical subscribers can resume after failover. `synchronized_standby_slots` ensures logical WAL senders do not advance past physical standbys. New `pg_replication_slots` columns: `invalidation_reason`, `inactive_since`, `synced`, `failover`. Synced slots cannot be consumed until promotion; `pg_sync_replication_slots()` errors if the standby lacks a `primary_slot_name`.
- **Restartpoint monitoring in `pg_stat_checkpointer`**: `restartpoints_timed`, `restartpoints_req`, `restartpoints_done` — the last shows how many actually completed (versus requested-but-skipped because the safe checkpoint record since the last restartpoint had not yet been replayed). This explains "spiky" `restartpoints_req` counts on standbys during heavy primary write bursts, which is normal and does not indicate resource pressure.
- **`pg_createsubscriber`**: converts a physical standby into a logical subscriber. Log messages for WAL recovery from backups added (Andres Freund). `pg_upgrade` now preserves logical replication slots.

#### PostgreSQL 18 (2025) — asynchronous I/O
- **AIO subsystem** (Andres Freund et al.): new `io_method` GUC — `worker` (default; dedicated I/O worker processes, count via `io_workers`, `Min: 1, Max: 32, Default: 3, Context: sighup`), `io_uring` (Linux, requires build `--with-liburing`), and `sync` (behaves like PG17, using `posix_fadvise` hints). Reads can be fetched directly into shared buffers rather than merely hinted to the OS page cache. Introspection via the new `pg_aios` view and `pg_stat_io`.
- **Scope**: AIO in PG18 covers **read** paths that know future block numbers. Per pganalyze's PG18 analysis, "As of the Postgres 18 Beta 1, asynchronous I/O is supported for sequential scans, bitmap heap scans, and maintenance operations like VACUUM"; per Neon, "Read operations only: Write operations, including WAL writes, remain synchronous." The recovery/redo read path is **not** an AIO consumer; WAL replay prefetching remains the PG15 posix_fadvise-based `xlogprefetcher`. `effective_io_concurrency` now directly controls internal async read-ahead depth (roughly `effective_io_concurrency × io_combine_limit`) when using an async `io_method`.
- DBA impact for recovery: AIO speeds ordinary query I/O and maintenance, but does not by itself accelerate crash/standby redo. Commit-latency bottlenecks (fsync) are unaffected — WAL write/flush remain synchronous.

#### PostgreSQL 19 (in development; Beta 1 released June 4, 2026)
Per the official Beta 1 announcement, "The PostgreSQL Project will release additional betas as required for testing, followed by one or more release candidates, until the final release around September/October 2026." All PG19 items below are from beta and may change before GA. Recovery-relevant confirmed items:
- **`pg_stat_recovery` view** (Xuneng Zhou, Shinya Kato; commits 01d485b14, 2d4ead6f4): a single-row view (returns no row when not in recovery) with `promote_triggered`, `last_replayed_read_lsn`, `last_replayed_end_lsn`, `last_replayed_tli`, `replay_end_lsn`, `replay_end_tli`, `recovery_last_xact_time`, **`current_chunk_start_time`** (diagnoses standbys stuck mid-record, with no prior function equivalent), and `pause_state`. It collapses the old sequence of `pg_last_wal_replay_lsn()` / `pg_get_wal_replay_pause_state()` / `pg_last_xact_replay_timestamp()` into one coherent reading for dashboards.
- **`WAIT FOR LSN` command** (Kartyshov Ivan, Alexander Korotkov, Xuneng Zhou; commits 447aae13b, 49a181b5d): a top-level utility with modes `standby_replay` (default), `standby_write`, `standby_flush`, and `primary_flush`, plus options `TIMEOUT` and `NO_THROW`. It enables read-your-writes consistency against replicas without application-side polling. It must be a top-level command (not in a function/procedure/DO block) and cannot hold a snapshot (so it is not usable above READ COMMITTED) — precisely the snapshot-deadlock problem that caused the PG17 attempt (`pg_wal_replay_wait()` procedure, commit 06c418e1, reverted via 772faafc) and the PG18 attempt (reverted via 3a7ae6b3d) to fail; neither shipped in a release. New wait events `WaitForWalFlush`/`WaitForWalReplay`/`WaitForWalWrite` were added.
- **Self-managing AIO worker pool** (Thomas Munro; commit d1c01b79d): per the Beta 1 announcement, "io_method=worker now automatically scales the number of I/O workers based on the new io_min_workers and io_max_workers settings," with additional `io_worker_idle_timeout` and `io_worker_launch_interval` GUCs replacing the static `io_workers`. Operational, not redo-specific.
- **Dynamic WAL level** (Masahiko Sawada; commit 67c20979c): with `wal_level=replica`, the effective level auto-increases to `logical` when the first logical slot is created and drops back at the next checkpoint after the last slot is dropped/invalidated; monitored via the read-only `effective_wal_level`. This interacts with slot synchronization: a separate change makes `sync_replication_slots=on` require `wal_level >= logical` (a `FATAL` otherwise), and whether `effective_wal_level` relaxes that requirement was still under discussion during beta.
- **AIO read-ahead scheduling improvements for large requests** (Andres Freund) — general, not redo-path-specific. There is **no evidence** in the v19 beta release notes/docs that `recovery_prefetch`/`xlogprefetcher` was reimplemented on the `read_stream` API or that AIO was wired into the WAL redo read path; it appears to remain posix_fadvise-based (verify against the v19 source tree if a definitive statement is needed).
- Related: `pg_sync_replication_slots()` now waits for synchronization completion (Ajin Cherian, Zhijie Hou; commit 0d2d4a0ec); `wal_sender_shutdown_timeout` added (Andrey Silitskiy, Hayato Kuroda); `pg_stat_wal_receiver.status` gains a `connecting` state (Xuneng Zhou); and hot-standby status can appear in the psql prompt via `%i` (Jim Jones). A recovery-durability bug was also fixed in the stable branches during this window: free-space-map changes during redo were not marked dirty with checksums enabled (Alexey Makhmutov), which could degrade standby performance after promotion.

## Recommendations
1. **Baseline your version's recovery model first.** On PG12+, purge legacy `recovery.conf` habits: put recovery/standby GUCs in `postgresql.auto.conf` (append order matters — last read wins), manage `standby.signal`/`recovery.signal` explicitly, and comment out `recovery_target_*` after a successful PITR to avoid re-shutdown loops (especially with `recovery_target_action=shutdown`, which does not remove `recovery.signal`).
2. **Cut recovery/failover time.** On PG14+ with many files, set `recovery_init_sync_method=syncfs` (Linux) after confirming you aren't sharing the filesystem with unrelated heavy writers. On PG15+ keep `recovery_prefetch=try` (default) and tune `maintenance_io_concurrency`/`wal_decode_buffer_size`; watch `pg_stat_recovery_prefetch` — if `skip_fpw` dominates while `prefetch` is high but real reads are few, the prefetcher is wasting fadvise calls (consider disabling it for that workload). Keep `checkpoint_timeout`/`max_wal_size` sized so post-crash redo is bounded.
3. **Protect `pg_wal` from slot-driven exhaustion.** Use `max_slot_wal_keep_size` (PG13+) as a hard backstop and monitor `pg_replication_slots` (`wal_status`, and `inactive_since` on PG17+). On PG17+ prefer replication slots plus `sync_replication_slots` for logical failover continuity, and set `synchronized_standby_slots` so logical senders don't outrun physical standbys.
4. **Instrument standby replay and conflicts.** Enable `log_recovery_conflict_waits` (PG14+). Monitor `pg_stat_io` (PG16+) for startup-process read patterns and `pg_stat_checkpointer` restartpoint counters (PG17+; trust `restartpoints_done`, not `restartpoints_req`). On PG19, adopt `pg_stat_recovery` for a single-view replay/lag/pause dashboard and consider `WAIT FOR LSN` for read-your-writes instead of application-side polling of `pg_last_wal_replay_lsn()`.
5. **Set correct async-I/O expectations.** On PG18/PG19, async I/O accelerates scans/VACUUM but not the WAL redo read path or commit fsync; do not expect `io_method=io_uring` alone to speed crash/standby recovery. On PG19, start `io_method=worker` with the auto-scaling defaults and only pin `io_max_workers` if you observe queueing.
6. **Thresholds that should change the plan:** if crash-recovery startup is dominated by the pre-replay sync phase (minutes), switch to `syncfs`; if standby replay lag (compare `pg_stat_recovery`/`pg_last_wal_replay_lsn` against the received LSN) grows under bursty primary writes, raise prefetch concurrency and verify restartpoints are completing (`restartpoints_done` advancing); if primary bloat rises with `hot_standby_feedback=on`, cap standby query duration (or use `max_standby_streaming_delay`) rather than disabling feedback.

## Caveats
- PG19 is in beta as of mid-2026; its release notes were marked provisional (dated "AS OF 2026-05-14," with Overview and Acknowledgments sections unfinished). Feature details, GUC defaults (e.g. `io_max_workers`), and the `sync_replication_slots`/`effective_wal_level` interaction may change before GA (around September/October 2026). Cited PG19 commit hashes are the short forms from the official release-note links; full commit-message bodies could not all be retrieved verbatim.
- The negative finding that PG18/PG19 async I/O does not cover the WAL redo read path is based on release notes, docs, and reputable community deep-dives (pganalyze, credativ, Neon, hgryoo.dev); a definitive statement would require a source-tree diff of `xlogprefetcher.c` between REL_18 and REL_19.
- Function names and file layout are stable across the versions covered, but line numbers in `xlogrecovery.c`/`xlog.c` and some helper-function boundaries drift between releases; treat file/function names as authoritative and any implied line references as illustrative.
- Where blog sources (Cybertec, pganalyze, EDB, dbi-services, Percona, Postgres Professional, InterDB) were used, claims were cross-checked against official documentation, release notes, and commit messages wherever possible; conflicts were resolved in favor of primary sources.
